--------------------------------------------------------------------------------
1. TUJUAN
--------------------------------------------------------------------------------

Dokumen ini menjelaskan sistem pengujian otomatis (automation QA) yang dibuat
khusus untuk target platform Web pada aplikasi Flutter SakuRapi.

Tujuan utama:
  - Memastikan seluruh alur fitur inti bekerja benar di browser (Chrome)
  - Mendeteksi regresi setelah perubahan kode sebelum deploy
  - Mendokumentasikan batasan pengujian web secara jujur
  - Menyediakan laporan pengujian yang dapat diulang kapan saja

Pengujian ini BUKAN pengganti pengujian manual di perangkat Android/iOS nyata.
Pengujian ini melengkapi pengujian manual untuk mempercepat siklus verifikasi.

--------------------------------------------------------------------------------
2. RUANG LINGKUP PENGUJIAN
--------------------------------------------------------------------------------

TERMASUK (IN SCOPE):
  - Rendering UI setiap layar utama
  - Validasi form (input kosong, nilai tidak valid, batas karakter)
  - Alur navigasi antar layar (bottom nav, tombol kembali, FAB)
  - State kosong (empty state) dan state error
  - Logika domain: perhitungan hutang, piutang, progress pembayaran
  - Integrasi provider Riverpod dengan fake repository (in-memory)
  - Format mata uang rupiah (Rp x.xxx)

TIDAK TERMASUK (OUT OF SCOPE):
  - Pengujian di perangkat Android/iOS fisik
  - Notifikasi lokal (platform channel tidak tersedia di web)
  - Google Sign-In nyata (memerlukan OAuth browser flow)
  - Persistensi data ke SQLite (Drift/WASM tidak diinisialisasi di unit test)
  - Sinkronisasi Firebase Firestore
  - Ekspor/impor file CSV (file picker adalah platform channel)
  - Kamera dan akses hardware

--------------------------------------------------------------------------------
3. PRINSIP UMUM PENGUJIAN
--------------------------------------------------------------------------------

A. ISOLASI PROVIDER
   Setiap widget test menggunakan ProviderScope dengan overrides ke
   FakeRepository (implementasi in-memory). Database nyata tidak pernah
   disentuh dalam widget test.

B. DETERMINISME
   Fixture data didefinisikan di test/helpers/test_helpers.dart agar
   setiap test menghasilkan output yang sama di semua mesin.

C. KEJUJURAN BATASAN
   Fitur yang bergantung pada platform channel (notifikasi, Google Sign-In,
   file picker) diberi status LIMITED atau SKIPPED. Tidak ada tes palsu
   yang pura-pura menguji fitur tersebut.

D. TIDAK ADA MOCK JARINGAN
   Tes tidak melakukan HTTP request nyata. Repository di-override dengan
   implementasi in-memory sehingga tes tidak memerlukan koneksi internet.

E. BAHASA UI
   Semua label, pesan validasi, dan teks yang dicari di tes menggunakan
   Bahasa Indonesia sesuai dengan AppStrings di kode produksi.

5. FITUR YANG TIDAK BISA DIUJI PENUH DI WEB
--------------------------------------------------------------------------------

A. GOOGLE SIGN-IN (LIMITED)
   Alasan: GoogleSignIn.signIn() melempar MissingPluginException di lingkungan
   test karena tidak ada platform channel yang terdaftar.
   Yang diuji: apakah dialog error ditampilkan (bukan crash aplikasi).
   Yang tidak diuji: alur OAuth, penerimaan token, simpan ke SharedPreferences.

B. NOTIFIKASI LOKAL (SKIPPED)
   Alasan: flutter_local_notifications menggunakan platform channel Android/iOS.
   Di web, plugin tidak tersedia sama sekali.
   Yang diuji: -
   Yang tidak diuji: penjadwalan, pengiriman, klik notifikasi.

C. EKSPOR/IMPOR CSV (SKIPPED)
   Alasan: share_plus dan file_picker adalah platform channel.
   Yang diuji: -
   Yang tidak diuji: pembuatan file, dialog berbagi, parsing CSV impor.

D. DATABASE DRIFT NYATA (LIMITED)
   Alasan: Drift/SQLite dengan WASM memerlukan inisialisasi khusus yang
   tidak tersedia di flutter test environment.
   Yang diuji: operasi CRUD lewat FakeRepository in-memory.
   Yang tidak diuji: migrasi schema, query SQL nyata, persistensi antar sesi.

E. FIREBASE FIRESTORE SYNC (SKIPPED)
   Alasan: memerlukan emulator Firebase atau koneksi internet nyata.
   Yang diuji: -
   Yang tidak diuji: push data, pull data, conflict resolution.

F. EMAIL LINK SIGN-IN — PENYELESAIAN (SKIPPED)
   Alasan: langkah "klik link di email" memerlukan deep link Android (app_links) atau
   redirect browser nyata. Tidak bisa disimulasikan di unit/widget test.
   Yang diuji: validasi form email, tampilan layar "link terkirim".
   Yang tidak diuji: alur klik link → buka app → sign-in selesai.

G. REALTIME MULTI-DEVICE SYNC (SKIPPED)
   Alasan: memerlukan dua perangkat fisik / emulator yang login akun yang sama,
   koneksi Firestore nyata, dan Firebase project yang sudah dikonfigurasi.
   Yang diuji: -
   Yang tidak diuji: propagasi perubahan Device A ke Device B, conflict resolution,
   perilaku listener saat offline/online.

9. URUTAN TES YANG DIJALANKAN SCRIPT
--------------------------------------------------------------------------------

Script: scripts/run_web_qa.sh

Langkah 1 — Persiapan
  Periksa dependensi: flutter, dart, google-chrome / chromium
  Buat direktori reports/web_qa/ jika belum ada
  Buat file log dengan timestamp

Langkah 2 — Analisis Statis
  flutter analyze
  Tujuan: tangkap type error, import rusak, warning sebelum menjalankan tes

Langkah 3 — Unit Test (non-UI)
  flutter test test/unit/
  Tujuan: verifikasi logika domain (kalkulasi hutang, piutang) secepat mungkin

Langkah 4 — Widget Test (semua)
  flutter test test/widget/ --platform chrome
  Tujuan: verifikasi UI render, validasi form, navigasi

Langkah 5 — Test Helper Verify
  flutter test test/helpers/ --platform chrome (jika ada test di sana)
  Tujuan: pastikan fake repository berfungsi benar

Langkah 6 — Integration Test (web)
  flutter test integration_test/web_flow_test.dart -d chrome
  Memerlukan: ChromeDriver berjalan di port 4444
  Tujuan: verifikasi alur end-to-end di browser

Langkah 7 — Ringkasan
  Hitung: jumlah PASS, FAIL, SKIPPED
  Tampilkan: total waktu, status akhir (LULUS / GAGAL)
  Simpan: semua output ke reports/web_qa/qa_<timestamp>.log

Langkah 8 — Exit Code
  0 = semua tes lulus
  1 = ada tes yang gagal (CI pipeline akan berhenti)

--------------------------------------------------------------------------------
10. ARTI STATUS TES
--------------------------------------------------------------------------------

  PASS
    Tes berhasil. Fitur berperilaku sesuai harapan di browser.
    Semua assertion terpenuhi. Tidak ada exception yang tidak ditangani.

  FAIL
    Tes gagal. Ada yang berubah dari perilaku yang diharapkan.
    Cek output di reports/web_qa/qa_<timestamp>.log untuk detail.
    Harus diperbaiki sebelum merge ke branch main.

  SKIPPED
    Tes sengaja dilewati karena fitur tidak dapat diuji di lingkungan ini.
    Biasanya karena ketergantungan platform channel (Android/iOS only).
    Status SKIPPED bukan kegagalan — ini keputusan desain yang disengaja.
    Fitur SKIPPED harus diuji manual di perangkat fisik.

  LIMITED
    Tes berjalan tapi hanya menguji sebagian dari fitur.
    Contoh: Google Sign-In — hanya menguji bahwa dialog error muncul,
    bukan alur autentikasi penuh.
    Perlunya pengujian manual tambahan untuk bagian yang LIMITED.

  ERROR
    Test runner sendiri gagal (bukan assertion tes yang gagal).
    Kemungkinan sebab: ChromeDriver tidak berjalan, dependensi hilang,
    kompilasi gagal, atau port konflik.
    Periksa: apakah ChromeDriver aktif? Apakah flutter build web berhasil?

--------------------------------------------------------------------------------
11. CARA MENJALANKAN
--------------------------------------------------------------------------------

Prasyarat:
  - Flutter SDK terinstall dan di PATH
  - Google Chrome terinstall
  - ChromeDriver (versi sama dengan Chrome) untuk integration test

Jalankan semua tes web:
  bash scripts/run_web_qa.sh

Jalankan hanya widget test:
  flutter test test/widget/ --platform chrome

Jalankan hanya unit test:
  flutter test test/unit/

Jalankan hanya integration test (ChromeDriver harus berjalan dulu):
  chromedriver --port=4444 &
  flutter test integration_test/web_flow_test.dart -d chrome

Lihat laporan terakhir:
  cat reports/web_qa/qa_<timestamp>.log

--------------------------------------------------------------------------------
12. CATATAN UNTUK PENGEMBANG SELANJUTNYA
--------------------------------------------------------------------------------

- Jika menambah fitur baru, tambahkan test di file yang relevan
  ATAU buat file baru di test/widget/
- Daftarkan test case baru di dokumen ini (bagian 6 atau 7)
- Jangan hapus tes yang SKIPPED — tes tersebut mendokumentasikan batasan nyata
- Untuk fitur yang menggunakan platform channel baru, tambahkan ke bagian 5
- Fixture data ada di test/helpers/test_helpers.dart — update jika schema berubah
- Jika schema database (Drift) berubah, FakeRepository mungkin perlu diperbarui
